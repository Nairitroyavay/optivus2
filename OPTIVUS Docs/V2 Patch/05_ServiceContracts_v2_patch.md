**Location:** At the very top of the doc, immediately after the title/audience/date

**Action:** INSERT new section.

```markdown
## Standard Request/Response Envelope

Every service method in this document uses the same envelope shape. v1 had drift
between methods — some returned bare data, some wrapped, some threw exceptions
for errors. v2 locks one pattern. Antigravity must follow it for every method,
including future methods not yet specified.

### Request envelope

Every service method accepts:

```typescript
interface ServiceRequest<TPayload> {
  request_id: string;       // UUIDv7 — generated by the caller
  user_id: string;          // UUID — extracted from auth token
  timestamp: string;        // TIMESTAMPTZ ISO-8601 — when the request was created
  schema_version: number;   // integer, currently 1
  payload: TPayload;        // method-specific input
}
```

`TPayload` is the only field that varies per method. Everything else is universal.

**Why each field:**
- `request_id`: enables tracing through the pipeline (event → rule log → AI message
  → notification all carry this ID). Without it, debugging "why did this user get
  this message?" is impossible at scale.
- `user_id`: explicit, never inferred. The auth layer extracts from JWT and passes
  through. Services trust this field — they don't re-validate the JWT.
- `timestamp`: client-stamped, server stores `received_at` separately. Difference
  reveals client clock skew, network latency, queue delays.
- `schema_version`: lets us evolve the envelope without breaking old callers.
  v1 → v2 envelope changes bump this.
- `payload`: the only method-specific shape. Strongly typed per method.

### Response envelope

Every service method returns:

```typescript
interface ServiceResponse<TData> {
  status: 'ok' | 'error';
  data: TData | null;       // present on ok, null on error
  error: ServiceError | null;  // present on error, null on ok
  latency_ms: number;       // server-computed, end-to-end
}

interface ServiceError {
  code: string;             // machine-readable, e.g. 'rate_limited', 'not_found'
  message: string;          // human-readable, never user-facing
  retryable: boolean;       // can the client safely retry?
  retry_after_seconds?: number;  // hint when retryable=true
  details?: Record<string, any>; // method-specific debug info
}
```

`status` is always either `ok` or `error`. There is no third state. There are no
exceptions thrown across service boundaries — failures become error responses.

### Contract rules

These rules apply to every service method. They are not optional.

**Rule 1 — Every API call emits an event.**
- Request received → `service.request_received` event
- Response sent → `service.response_sent` event
- Error → `service.error` event
- These are low-priority operational events, not coaching-relevant. They power
  observability and rate limiting.

**Rule 2 — Every state mutation is tracked via event.**
- Creating, updating, or deleting any data row emits a domain event
  (`habit.created`, `task.completed`, etc.) in addition to the operational events
  above.
- Read operations do not emit events.

**Rule 3 — `request_id` propagates through the entire pipeline.**
- Service receives request with `request_id`
- Any events emitted include `metadata.request_id`
- Any AI rule logs reference the originating `request_id`
- Any coach messages or notifications generated trace back to `request_id`

**Rule 4 — Errors are categorized, not free-form.**
- The `error.code` field is from a fixed enum (see error code catalog below).
- Adding a new code requires updating the catalog.
- Free-form error strings go in `error.message`, never in `error.code`.

**Rule 5 — `retryable` is honest.**
- `retryable: true` means: the client can safely retry with the same `request_id`
  and the system will handle deduplication.
- `retryable: false` means: the request will fail again. Don't retry.
- Lying about retryability creates retry storms and duplicate side effects.

### Error code catalog

These are the canonical error codes. Use these, not custom strings.

| Code | When | Retryable | Notes |
|---|---|---|---|
| `not_authenticated` | No or invalid auth token | false | Client must reauth |
| `not_authorized` | Auth valid but insufficient permission | false | |
| `not_found` | Resource doesn't exist | false | |
| `validation_failed` | Request payload malformed | false | `details` lists the bad fields |
| `rate_limited` | Too many requests | true | `retry_after_seconds` set |
| `conflict` | State change conflicts (e.g. concurrent edit) | true | Client should refetch and retry |
| `dependency_unavailable` | Downstream service down (LLM, DB, etc.) | true | `retry_after_seconds` typically 5-30 |
| `quota_exceeded` | User over their daily budget (tokens, messages, etc.) | false | Different from rate_limited — this is a per-user cap |
| `internal_error` | Unhandled server error | true | Logged with full stack trace |
| `feature_disabled` | Feature is gated and not available for this user | false | |

### Example: TaskService.completeTask

**Request:**
```json
{
  "request_id": "01927f3a-1234-7000-8000-aaaa11110001",
  "user_id": "01927f3a-1234-7000-8000-000000000001",
  "timestamp": "2026-04-26T14:23:00+05:30",
  "schema_version": 1,
  "payload": {
    "task_id": "01927f3a-1234-7000-8001-cccc20020003",
    "completion_notes": "Done in 25 min — went well"
  }
}
```

**Response (ok):**
```json
{
  "status": "ok",
  "data": {
    "task_id": "01927f3a-1234-7000-8001-cccc20020003",
    "new_state": "completed",
    "completed_at": "2026-04-26T14:23:01+05:30",
    "events_emitted": ["task.completed"]
  },
  "error": null,
  "latency_ms": 47
}
```

**Response (error — task already completed):**
```json
{
  "status": "error",
  "data": null,
  "error": {
    "code": "conflict",
    "message": "Task is already in 'completed' state — cannot transition again",
    "retryable": false,
    "details": {
      "current_state": "completed",
      "completed_at": "2026-04-26T13:55:00+05:30"
    }
  },
  "latency_ms": 12
}
```

### Why this matters

A typed envelope means:
- Antigravity generates consistent code across services without inventing patterns
- Logging is uniform — every log line has `request_id`, `user_id`, `latency_ms`
- Retry logic on the client is one wrapper, not per-method
- Error handling on the client is one switch statement on `error.code`, not
  per-method exception types
- Testing is uniform — every method's test suite has the same shape

A free-form contract means: every service grows its own way, and 6 months later
you have 47 different error patterns. v1 was sliding toward that. v2 stops it.
```

## Patch 2 of 2 — Rewrite all method signatures using the envelope

**Location:** Every existing service method definition in the v1 doc.

**Action:** REPLACE each method signature using the envelope pattern. Below is
the rewrite pattern with one full example.

### The rewrite pattern

For every existing v1 method:

**v1 form (was):**
```typescript
class TaskService {
  Future<Task> completeTask(String taskId, String? notes);
}
```

**v2 form (becomes):**
```typescript
class TaskService {
  Future<ServiceResponse<CompleteTaskResponseData>> completeTask(
    ServiceRequest<CompleteTaskPayload> request
  );
}

interface CompleteTaskPayload {
  task_id: string;
  completion_notes?: string;
}

interface CompleteTaskResponseData {
  task_id: string;
  new_state: TaskState;
  completed_at: string;
  events_emitted: string[];
}
```

### Apply this pattern to every method in the v1 doc

The v1 doc has methods on these services (from audit + ServiceContracts v1):

| Service | Methods to rewrite |
|---|---|
| **EventService** | emit, query, subscribe, replay |
| **TaskService** | createTask, startTask, pauseTask, completeTask, abandonTask, listTasks |
| **HabitService** | createHabit, logHabit, listHabits, archiveHabit |
| **StreakService** | getStreak, updateStreakOnDayClose, breakStreak, recoverStreak |
| **RoutineService** | createRoutine, updateRoutine, recordCompletion, getCompletionsForDate |
| **NotificationService** | scheduleNotification, cancelNotification, markRead, markTapped, rescheduleAfterReinstall |
| **CoachService** *(new in v2)* | onEvent, generateMessage, queryHistory |

For each method:
1. Define the payload type (the v1 method's parameters become a payload struct)
2. Define the response data type (the v1 method's return type becomes the data field)
3. Wrap method signature in `ServiceRequest<...>` / `ServiceResponse<...>`
4. List error codes the method may return (from the catalog above)

### Worked example: HabitService.logHabit

**v1 (assumed):**
```dart
Future<HabitLog> logHabit(String habitId, double quantity, String? trigger);
```

**v2:**
```typescript
class HabitService {
  Future<ServiceResponse<LogHabitResponseData>> logHabit(
    ServiceRequest<LogHabitPayload> request
  );
}

interface LogHabitPayload {
  habit_id: string;
  occurred_at: string;       // ISO-8601, may be backdated
  quantity?: number;
  trigger?: 'stress' | 'boredom' | 'social' | 'habit' | 'craving' | string;
  note?: string;
  mood_score?: 1 | 2 | 3 | 4 | 5;
}

interface LogHabitResponseData {
  habit_log_id: string;
  habit_id: string;
  occurred_at: string;
  events_emitted: string[];  // typically ['habit.good_logged' OR 'habit.bad_logged']
  streak_after: {
    current_streak: number;
    longest_streak: number;
    state: 'active' | 'broken' | 'recovering';
  };
}

// Possible error codes:
// - validation_failed (bad habit_id, future occurred_at)
// - not_found (habit_id doesn't exist for this user)
// - conflict (duplicate log within idempotency window)
// - dependency_unavailable (Postgres or Redis down)
// - internal_error
```

### Apply pattern, don't copy method-by-method

Antigravity: rather than rewriting every v1 method individually here (which would
double the doc size), apply the pattern in §1 to every existing method when
generating service code. The pattern is unambiguous. The error code catalog is
fixed. The envelope is universal.

If a method's payload or response shape isn't obvious from the v1 description,
flag it in implementation and ask Nairit before guessing.
