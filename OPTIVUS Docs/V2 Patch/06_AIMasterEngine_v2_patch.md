## AI Pipeline Hard Rules

Three rules govern every coach message Optivus produces. These are non-negotiable.
If implementation conflicts with any of them, the implementation changes — not the
rule.

### Hard Rule 1 — AI is the last step in the pipeline, never the first

The Decision Engine, Rule Matcher, Priority Filter, and Context Builder all run
before any LLM call. If any of them produces no eligible message, no LLM call
happens. The LLM never decides whether to speak — it only generates wording for
a decision that has already been made.

**What this prevents:** Antigravity (or any future engineer) calling the LLM with
"here's the user's data, decide if you should say something" and routing on the
LLM's output. That pattern is unbounded, expensive, and unauditable. We don't do
it. Period.

**What this enables:** the engine remains testable without the LLM. Phase 1 of
the build (per §11) is a fully working coach using only fallback messages — no
LLM at all. The LLM is enhancement, not dependency.

### Hard Rule 2 — Every message passes the Output Validator before reaching the user

After the LLM responds (or after fallback selection), the message goes through a
validator stage that checks tone, repetition, and emotional overload. Failures
fall back to `rule.fallback_message`. Failures of the fallback drop the message
entirely.

This rule is detailed in §7.6 (Output Validator) below.

### Hard Rule 3 — The system functions without AI

If Claude is down, rate-limited, or over budget:
- The Decision Engine still runs
- Rules still match
- Priority filter still selects
- Context Builder still builds
- The user gets `rule.fallback_message` with placeholders filled in
- Notifications still fire
- Streaks still update
- Day-close still happens

The AI is the *voice* of Optivus. The *brain* is rule-based and works on its own.
A user who only ever sees fallback messages still gets a coherent product.

This rule is enforced by Phase 1 of the build (Engine + fallbacks, no LLM) being
shippable on its own. If Phase 1 produces a working coach experience, Hard Rule 3
is satisfied. If Phase 1 produces a broken experience, the architecture violates
the rule and must be fixed before Phase 2.
```

```markdown
# Part 6 — Memory Model

The coach needs context. v1 had `ContextSnapshot` for short-term context but said
nothing about long-term. This section closes that gap, with a hard constraint:
all memory is user-readable and user-deletable. No vector embeddings in v1.

## 6.1 Two-tier memory

| Tier | Time horizon | Storage | What it contains |
|---|---|---|---|
| **Short-term** | Last 7 days | Postgres `events` table (current month partition) + `events_recent` mirror in Firestore for fast streams | Full event log, full fidelity |
| **Long-term** | 8 days to ∞ | Postgres `ai_context_snapshots` table | Weekly summarized rows in human-readable English |

Short-term is read on every coach decision (Context Builder builds the snapshot
from it). Long-term is read only when constructing a "broader picture" prompt —
crisis context, weekly review, identity drift detection.

## 6.2 Long-term: summarized rows, not embeddings

The decision: long-term memory is stored as **summarized natural-language rows**,
not as vector embeddings.

**The rejected alternative:** vector embeddings of every event or chat message,
stored in pgvector or a dedicated vector DB, retrieved by semantic similarity.

**Why rejected:** embeddings are not user-readable. A user cannot look at a row
and say "delete that — I don't want the AI to remember it." This violates the
Data Ownership rule in PRD §[Data Ownership]. Even with full account deletion
clearing the embeddings, the user can't selectively prune memory between deletions.

**The chosen approach:** one summarized row per user per week, generated by a
weekly Cloud Scheduler job.

### Summary row structure

```
ai_context_snapshots row (per Database Schema §3.10):
{
  id: UUID,
  user_id: UUID,
  occurred_at: TIMESTAMPTZ (the date this summary covers, end-of-week),
  snapshot: JSONB {
    "type": "weekly_summary",
    "week_starting": "2026-04-19",
    "completion_pct_avg": 0.65,
    "completion_pct_trend": "declining",
    "habits": {
      "cigarettes": {"count": 3, "trend": "stable"},
      "gym": {"completions": 2, "target": 4, "trend": "down"}
    },
    "user_state_during_week": ["on_track","on_track","slipping","slipping","slipping","relapsing","slipping"],
    "coach_messages_count": 4,
    "coach_topics_covered": ["cigarettes","gym","stress"],
    "stress_markers_in_chat": 1,
    "crisis_markers_in_chat": 0,
    "summary_text": "Week of Apr 19: completion declined from 80% to 50%. Smoked 3 days. Mentioned 'overwhelmed' once. Coach spoke 4 times, all C-rule (slip)."
  },
  chat_session_id: NULL (weekly summaries are not chat-bound),
  schema_version: 1
}
```

The `summary_text` field is what the AI reads when fetching long-term context.
The structured fields (`habits`, `user_state_during_week`, etc.) are for filtering
and analytics — the AI doesn't see them directly.

### Generation

Cloud Scheduler runs every Sunday at 23:55 user-local time. For each user:

1. Aggregate the past 7 days from events table
2. Compute statistical fields (averages, trends, counts)
3. Call Claude Haiku once with a structured prompt: "Given these stats and
   coach interactions, write a 2-sentence summary of the user's week"
4. Insert one row into `ai_context_snapshots`

Cost: one Haiku call per user per week. Cheap.

If Claude is unavailable during weekly generation, a deterministic template
fallback fills `summary_text` ("Week of Apr 19: completion 65%, declining trend.
Smoked 3 days."). No retry — next week's generation will catch up.

## 6.3 User control surface

In Profile → Coach Memory, the user sees:

- A list of every weekly summary row, newest first
- Each row shows: week range, summary_text, stat highlights
- Each row has a "Delete" button — one tap, irreversible
- A "Delete all memory" button at the top — clears all rows

When the user deletes a row, it's removed immediately from the database. The
Context Builder, on its next read, will not see it. There is no cache to invalidate
beyond Redis `coach:{user_id}:context_summary` (5-min TTL — natural eviction).

## 6.4 What the AI sees

When the Context Builder constructs a snapshot for a coach decision:

- For most decisions: short-term only (last 7 days from events)
- For weekly review messages: short-term + last 4 long-term rows
- For crisis-aftermath context: short-term + last 12 long-term rows
- For identity drift detection: short-term + last 24 long-term rows

The number of long-term rows pulled is per-rule, defined in `rule.long_term_context_weeks`.
Most rules don't need long-term context. Strategic AI Planner does.

## 6.5 What the AI never stores

To enforce the "no hidden memory" rule:

- LLM provider's request caching (`anthropic-cache-control`) is set to disabled
  for coach calls. We don't want Claude's infrastructure caching user context.
- No fine-tuning on user data.
- No "conversation memory" features beyond the explicit `chat_session_id` grouping
  for in-app chat continuity (which is itself stored in user-readable Postgres rows).
- Logs of LLM calls (`coach_speak_log` per §10) include the prompt sent and the
  message returned, but the user can audit and delete these via the same Coach
  Memory surface.

Memory is what's in the database. The database is what the user owns. There is
no third place.
```

```markdown
## 7.6 Output Validator

Between LLM response and message delivery, every output passes through three
checks. Failures fall back. Failures of fallback drop the message.

### Check 1 — Tone (anti-pattern regex)

The §10.2 anti-pattern list is enforced as a regex post-check on every LLM output.

```dart
final ANTI_PATTERN_REGEX = RegExp(
  r'^I (notice|noticed|see)\b'                          // banned opener
  r'|are you (feeling|sad|stressed|frustrated)\b'       // assumes emotion
  r'|you (should|need to|must|have to)\b'               // commands
  r"|let'?s (crush|smash|conquer)\b"                    // fake hype
  r'|take a deep breath\b'                              // therapy-speak
  r"|don'?t be hard on yourself\b"                      // patronizing
  r'|it'?s important that you\b'                        // soft command
  r'|❤️|💪|🙏'                                            // forbidden emojis (config-driven)
  ,
  caseSensitive: false,
);

bool failsToneCheck(String message) {
  return ANTI_PATTERN_REGEX.hasMatch(message);
}
```

If the message matches: fall back to `rule.fallback_message`. Log the failure with
the matched pattern to `coach_speak_log.tone_check_failures`. Track these — if a
specific pattern is causing >5% fallback rate, the system prompt needs tuning.

### Check 2 — Repetition

Every coach message is fingerprinted. Before sending a new message, check the
last 20 coach messages to this user.

```dart
double similarity(String a, String b) {
  // Levenshtein distance, normalized to 0-1
  final distance = levenshtein(a.toLowerCase(), b.toLowerCase());
  final maxLen = max(a.length, b.length);
  return 1.0 - (distance / maxLen);
}

bool failsRepetitionCheck(String newMessage, List<String> recentMessages) {
  for (final old in recentMessages) {
    if (similarity(newMessage, old) > 0.85) return true;
  }
  return false;
}
```

If similarity > 0.85 against any of the last 20: fall back. Log to
`coach_speak_log.repetition_check_failures`.

The 0.85 threshold is calibrated to allow stylistic variation ("Four cigarettes
today" vs "That's the fourth today") but block near-duplicates ("Four cigarettes
today. Want to talk?" vs "4 cigarettes today. Want to talk it through?").

If even the fallback fails repetition (the user has hit fallback 20 times in a row
on the same rule — implies a stuck pattern), suppress the message entirely and
log to `coach_speak_log.repetition_lockout`. The Decision Engine should detect
this state via its cooldowns; if it's still happening, there's a bug to fix.

### Check 3 — Emotional overload

If the coach has spoken 4+ times in the last 6 hours about negative-priority
topics (slips, missed routines, bad-day, stress), and this new message is also
negative-priority, suppress it.

```dart
bool failsEmotionalOverloadCheck(
  ServiceMessage newMessage,
  List<CoachMessageLog> last6h,
) {
  if (newMessage.priority > 2) return false;  // only applies to priority 1-2 negative msgs
  if (newMessage.archetype == 'celebratory') return false;  // celebrations always allowed
  if (newMessage.rule_id == 'crisis_E1' || newMessage.rule_id == 'crisis_E2') {
    return false;  // crisis bypasses everything
  }

  final negativeCount = last6h.where((m) =>
    m.priority <= 2 && m.archetype != 'celebratory'
  ).length;

  return negativeCount >= 4;
}
```

If overloaded: drop the message (do not fall back — fallback would be another
negative message). Log to `coach_speak_log.emotional_overload_drops`.

The user gets silence instead of pile-on. Silence is a feature (Principle 10
in §10).

### Validator sequence

```
LLM returns string
  → Check 1 (tone) → if fails → use rule.fallback_message
  → Check 1 again on fallback → if fails → drop, log, alert engineering
  → Check 2 (repetition) → if fails → use rule.fallback_message
  → Check 2 again on fallback → if fails → drop, log
  → Check 3 (emotional overload) → if fails → drop (no fallback)
  → All checks pass → deliver to user
```

Latency budget for all three checks combined: under 50ms. They run on-device after
the LLM response arrives. Network is not involved.

### Why this validator exists

Three failure modes the v1 design didn't catch:

- **Tone drift over many calls:** the LLM, even with system prompts, occasionally
  produces "I noticed..." or "Are you feeling..." — these regex checks catch them
  before the user sees them.
- **Repetition fatigue:** users notice when the coach sounds the same. Even
  varied wording from the LLM can produce near-duplicates over weeks.
- **Pile-on cascade:** five small slips in one afternoon, each producing a coach
  message, makes the coach feel relentless. Emotional-overload check is the
  circuit breaker.

The validator is small but high-leverage. It's the difference between a coach that
feels human and a coach that feels like a chatbot.
```

### Case E — Crisis (priority 1, special, multi-tier)

v1 specified a single message + 24h silence. Real users in crisis don't always
call the number. v2 adds escalation tiers and a manual human-handoff path.

The tier the user lands in depends on prior crisis triggers. A user's first
crisis trigger in 7 days is Tier 1; a second within 7 days escalates to Tier 2;
a third within 14 days escalates to Tier 3.

#### RULE E1 — Tier 1 (first trigger in 7 days)

```
WHEN chat.user_message contains §6.2 CRISIS_MARKERS_HIGH
  AND prior_crisis_triggers_in_7d == 0
DO speak immediately, ignore all gating
WITH archetype 'crisis' (special, see §10)
SAY: fixed, non-LLM-generated message, exact wording:
     "I'm hearing something heavy. I'm an app — I can sit with you, but I want
      you to talk to someone real too. iCall (India): 9152987821. iCall is free
      and confidential. Want me to keep this thread quiet so you have a safe
      space here?"
SUGGEST [Open iCall info], [Stay here, just listen]

POST-MESSAGE:
  Disable all proactive coaching for next 24h
  Coach goes into "listening mode" — only responds to user-initiated chat
  Re-enable after 24h OR if user types "I'm ok" / "I want my coach back"
  Increment user.crisis_trigger_count_7d
```

#### RULE E2 — Tier 2 (second trigger within 7 days)

```
WHEN chat.user_message contains CRISIS_MARKERS_HIGH
  AND prior_crisis_triggers_in_7d >= 1
DO speak immediately, ignore all gating
WITH archetype 'crisis'
SAY: fixed message, escalated:
     "I'm worried about you. This is the second time today/this week we've been
      here together. Please call iCall: 9152987821 or reach out to someone you
      trust today — a friend, family, anyone. I'll be here when you come back."
SUGGEST [Open iCall info], [Send a message to someone], [Stay here, just listen]

POST-MESSAGE:
  Silence lock extended to 72 hours (proactive coach silence)
  Internal alert: founder/ops Slack notification
       Alert content: { user_id, tier: 2, timestamp }
       NEVER includes message content or PII beyond user_id
  Increment user.crisis_trigger_count_7d
```

#### RULE E3 — Tier 3 (third trigger within 14 days)

```
WHEN chat.user_message contains CRISIS_MARKERS_HIGH
  AND prior_crisis_triggers_in_14d >= 2
DO speak immediately, ignore all gating
WITH archetype 'crisis'
SAY: fixed iCall referral PLUS a draft offer:
     "I'm here. I want to make this one step easier. Want me to help you draft
      a message to someone in your life right now? Even just 'I'm struggling and
      need to talk' is enough to start."
SUGGEST [Help me draft a message] [Open iCall info] [Stay here, just listen]

POST-MESSAGE:
  Silence lock 72 hours
  Internal alert: HIGH-PRIORITY operational notice
       Alert content: { user_id, tier: 3, timestamp }
       Goes to founder + on-call rotation
  Crisis SOP triggered (see §10.3)
  Draft starter ("I've been struggling and need to talk") populated in chat composer
  Increment user.crisis_trigger_count_14d

`crisis_trigger_count_7d` decrements after 7 days clean.
`crisis_trigger_count_14d` decrements after 14 days clean.
A user who has 14+ days clean returns to Tier 1 territory.

#### Privacy boundary

Internal alerts contain user_id only. Never:
- Message content
- User's name, email, phone
- Any PII

14:23 IST." They then have a defined SOP (§10.3) to follow. They do not see what
the user wrote.

| Tier 1 (1st in 7d) | iCall msg | No | 24h |
| Tier 2 (2nd in 7d) | Stronger msg | Slack | 72h |

## 10.3 Crisis SOP (manual handoff)

When Tier 3 fires, an on-call human (founder in v1, support team later) follows
this SOP within 24 hours.

### Step 1 — Verify the alert
- Check `ai_rule_logs` for the user_id
- Confirm Tier 3 trigger is real, not a test or duplicate
- If duplicate (same user already actioned in last 24h): no action

### Step 2 — Send a manual, non-templated email
- Subject: "Hey — checking in" (literal subject, no exclamation)
- Body: kind, short, non-prescriptive. Examples:
  > "Hey [name],
  >
  > I noticed you've been having a rough time using Optivus. I'm one of the
  > people who built it. I'm not writing as the app — I'm writing as a person.
  >
  > If you ever want to talk to a real human, here are some options:
  > - iCall: 9152987821 (free, confidential, English/Hindi)
  > - Vandrevala Foundation: 1860-2662-345 (24/7 helpline)
  > - Or just write back. I read every reply.
  >
  > No pressure. I just wanted you to know that the app on your phone isn't your
  > only option."
- DO NOT use a template. Each email is written by a human.
- DO NOT include screenshots, message logs, or anything from inside the app.

### Step 3 — Log the action
- Write to `crisis_handoffs` Postgres table:
  ```
  { user_id, alert_received_at, action_taken_at, action_type: 'manual_email',
    notes (free text from the operator) }
  ```
- This log is never shown to the user. It's the operational audit trail.

### Step 4 — Wait
- Do not follow up unless the user replies
- If the user replies: respond as a human, escalate to actual mental health
  resources if needed
- If the user uninstalls the app: do not contact again

### What this SOP is NOT

- Therapy. We are not therapists.
- Surveillance. The user knows from onboarding that crisis triggers can result
  in human outreach.
- Marketing. No "by the way, have you tried X feature."
- Automated. Every email is a human sending a human message.

### Escalation past SOP

If the operator observes (in the user's chat history with the AI, accessed only
during SOP) explicit imminent danger ("I'm going to take pills tonight"),
escalation to local emergency services may be appropriate. This is not a routine
SOP step — it requires manual judgment and consultation.

For India v1: AASRA (24/7 suicide prevention) at +91-22-27546669. Local police
108 if imminent danger and user has shared location voluntarily in the chat.

### Why a manual SOP, not an automated one

Two reasons. First: automated mental health intervention beyond an iCall referral
does more harm than good. Real humans should respond to real human distress.
Second: at Optivus's current scale (solo dev, low MAU), the volume is small
enough that a manual SOP is feasible. When the volume grows past what a founder
can personally handle, the SOP becomes "hire a part-time crisis-trained person
to do this," not "automate it further."

# Part 11 — Cost Controls

The schema doc projects ~$1,200/month for Anthropic API at 100k MAU. That's the
largest variable cost line and v1 had no runtime control on it. v2 adds caps.

## 11.1 Per-user daily token budget

Every user has a daily token cap, tracked in Redis.

```
Key: ai_tokens:{user_id}:{YYYY-MM-DD}
Value: integer (sum of input + output tokens consumed by coach today)
TTL: 26 hours (covers timezone slippage)
```

**Default cap:** 4,000 input + 800 output tokens per day. Roughly equivalent to
~10 coach messages with full context.

**Enforcement:**
- Before each LLM call: check current consumption against cap
- If at or above cap: skip LLM call, use `rule.fallback_message`
- Increment counter atomically with `INCRBY` after each call
- The user sees no special indicator — fallback messages are designed to be
  coherent on their own

**Cap is per accountability mode:**

| Accountability | Daily token cap |
|---|---:|
| Forgiving | 2,500 (matches 4 messages/day budget) |
| Strict | 4,000 (matches 7 messages/day budget) |
| Ruthless | 6,000 (matches 10 messages/day budget) |

These match the speak budget from §1.5. If a user hasn't exhausted their speak
budget but has exhausted their token budget, the next message uses fallback.

## 11.2 Global monthly cap

A hard ceiling on total Anthropic spend for the month, enforced server-side.

**Default cap:** $1,500/month (configurable per environment).

**Tiered response as cap fills:**

| Spend % | Action |
|---|---|
| <60% | Normal operation |
| 60-80% | Alert in Slack, no behavior change |
| 80-95% | Switch all NEW users to fallback-only mode (existing users unaffected). Alert escalated to founder. |
| 95-100% | Switch ALL users to fallback-only mode. P0 alert. |
| >100% | Should never happen — hard kill switch already engaged at 100% |

**How the kill switch works:**
- Redis key: `ai:global_kill_switch` (boolean, default false)
- When monthly spend crosses 100%: set to true
- Every LLM call site checks this key first. If true: skip LLM, use fallback.
- Resets on the 1st of each month at 00:00 UTC (or earlier if spend was raised
  via deployment).

## 11.3 Per-call cost monitoring

Every LLM call records to `coach_speak_log`:

- `llm_tokens_in` — input tokens
- `llm_tokens_out` — output tokens
- `llm_estimated_cost_usd` — computed at log time using current pricing
- `llm_model` — which model was used (Haiku 4.5 default; Haiku 3 fallback)

A daily aggregation job rolls these up:

```sql
-- Run daily, results piped to monitoring dashboard
SELECT
    DATE(created_at) AS day,
    llm_model,
    COUNT(*) AS calls,
    SUM(llm_tokens_in) AS total_input,
    SUM(llm_tokens_out) AS total_output,
    SUM(llm_estimated_cost_usd) AS total_cost
FROM coach_speak_log
WHERE created_at > NOW() - INTERVAL '30 days'
GROUP BY 1, 2
ORDER BY 1 DESC, 4 DESC;
```

This aggregation drives the global cap enforcement.

## 11.4 Model fallback chain

If primary model fails or is over budget, a chain of fallbacks:

1. Claude Haiku 4.5 (primary — cheap, fast, persona-disciplined)
2. Claude Haiku 3 (fallback — cheaper, slightly weaker)
3. Local fallback message (no LLM)

Step 2 is engaged when:
- Anthropic returns 5xx three times in a row for the same user
- Budget is between 80-95% of monthly cap (cost reduction mode)

Step 3 is engaged when:
- Both Haiku models fail
- Budget is over 95% of monthly cap
- User token budget is exhausted

## 11.5 Cost ceiling configuration

These constants live in code, not in Firestore. Changing them requires a deploy.

```dart
const PER_USER_DAILY_TOKEN_CAP = {
  'forgiving': 2500,
  'strict': 4000,
  'ruthless': 6000,
};

const GLOBAL_MONTHLY_CAP_USD = 1500;
const COST_REDUCTION_THRESHOLD_PCT = 80;
const KILL_SWITCH_THRESHOLD_PCT = 95;
```

These defaults are placeholders for v1 launch. The real numbers depend on:
- Runway (how much can the project absorb in API spend?)
- MAU (the per-user economics)
- Willingness to subsidize (early users typically subsidized to gather feedback)

A finance review should set the production numbers. Until then, defaults err
on the side of caution — easier to raise caps than to recover from a bill shock.

## 11.6 Why this matters

A coach product without cost controls is a financial unknown. One viral signup
wave or one bug in the rule engine that fires too often can multiply spend 10x
in days. The caps in this section turn unknown cost into known cost.

The product also remains fully functional under cap exhaustion — users in
fallback-only mode get coherent rule-based messages. They lose the LLM-varied
wording but not the coach. This is Hard Rule 3 (system functions without AI)
in action.
```
